# Shopify Conventions

## Supersymmetry Theme

This document defines how Shopify Online Store 2.0 features should be implemented in this repository.

These conventions are mandatory.

When Shopify provides an official solution, always prefer it over a custom implementation.

---

# Core Principles

Follow Shopify conventions.

Avoid fighting the platform.

Avoid recreating platform functionality.

Build with Shopify, not around Shopify.

---

# Theme Editor First

Every feature should be designed for merchants first.

If a merchant is likely to want to customize something, expose it through Theme Settings rather than hardcoding it.

Good candidates:

- headings
- labels
- icons
- spacing
- colors (where appropriate)
- promotional copy
- shipping thresholds
- announcements
- feature toggles

Do not expose settings that add complexity without real value.

A smaller, better-organized Theme Editor is preferable to one with excessive options.

---

# Dynamic Sources

Whenever a setting represents product-specific content, support Dynamic Sources.

Examples:

✓ Product subtitle

✓ Materials

✓ Fit notes

✓ Delivery notes

✓ Care instructions

✓ Badges

Do not create Theme Settings for values that belong to individual products.

---

# Metafields

Use metafields for content that varies by product, collection, article, or page.

Examples:

- Fit label
- Material composition
- Country of manufacture
- Editorial subtitle
- Delivery note
- Care instructions
- Feature icons
- Lifestyle notes

Never encode this information in JavaScript.

Never duplicate metafield values in Theme Settings.

---

# Theme Settings

Theme Settings are for global behavior.

Examples:

- Free shipping threshold
- Cart drawer options
- Animation preferences
- Global announcements
- Icon styles
- Button styles
- Typography
- Spacing

Theme Settings should not contain product data.

---

# Section Responsibilities

Sections own:

- layout
- block rendering
- schema
- merchant customization

Sections should not own reusable UI.

Move reusable components into snippets.

---

# Snippet Responsibilities

Snippets own presentation.

Examples:

- Button
- Icon
- Badge
- Price
- Cart item
- Product card
- Shipping progress
- Delivery estimate

Snippets should avoid owning business logic.

---

# Blocks

Blocks exist to give merchants flexibility.

Do not create blocks simply because Shopify supports them.

Use blocks when merchants genuinely benefit from rearranging content.

Avoid unnecessary nesting.

---

# Templates

Templates should compose sections.

Templates should remain thin.

Avoid logic.

Avoid duplication.

---

# Assets

JavaScript belongs in assets.

CSS belongs in assets.

Avoid inline JavaScript.

Avoid inline CSS except when generated by Shopify settings.

---

# JavaScript

JavaScript should progressively enhance the storefront.

The storefront must remain functional without JavaScript whenever possible.

Do not require JavaScript for basic purchasing.

---

# AJAX

Use Shopify AJAX APIs.

Avoid custom endpoints.

Preferred APIs:

- cart.js
- cart/add.js
- cart/change.js
- cart/update.js
- bundled section rendering

Do not reload the page unless absolutely necessary.

---

# Section Rendering API

Whenever cart, product, or section content changes dynamically, prefer Shopify Section Rendering instead of manually reconstructing HTML.

Benefits:

- keeps Liquid as the rendering source
- preserves app compatibility
- preserves localization
- avoids duplicated templates

---

# Product Forms

Always use Shopify product forms.

Do not invent custom purchase flows.

Preserve compatibility with:

- selling plans
- line item properties
- subscriptions
- gift cards
- bundles

---

# Variant Selection

Variant changes should update:

- URL
- price
- availability
- gallery
- sticky ATC
- inventory
- pickup availability

Maintain a single source of truth.

---

# Cart

Use one cart implementation.

Never maintain:

drawer cart

mini cart

popup cart

page cart

as separate systems.

They should consume the same cart state.

---

# Cart State

Cart should update:

- header badge
- drawer
- totals
- shipping progress
- checkout button
- quantity controls

from one source.

---

# Localization

Never hardcode customer-facing text.

Always use locale files.

Support pluralization.

Support interpolation.

Support translations.

---

# Money

Never manually format currency.

Always use Shopify money filters.

Never concatenate currency symbols.

---

# Images

Use Shopify image filters.

Request appropriately sized images.

Avoid oversized downloads.

Always define width and height where practical to reduce layout shift.

Support responsive images.

Prefer lazy loading below the fold.

---

# Video

Prefer native Shopify media support.

Do not embed third-party players unnecessarily.

---

# Icons

Use one icon system.

Avoid mixing icon libraries.

SVG preferred.

Inline SVG when appropriate.

---

# Buttons

Maintain one reusable button component.

Avoid multiple button implementations.

Support:

- primary
- secondary
- tertiary
- loading
- disabled

through modifiers rather than duplication.

---

# Forms

Every form should:

- validate gracefully
- support autofill
- remain keyboard accessible
- provide useful errors

---

# Theme App Extensions

Never modify app block output.

Respect app boundaries.

Allow apps to coexist with theme functionality.

---

# CSS Variables

Use CSS custom properties for design tokens.

Examples:

spacing

radius

colors

typography

animation

Avoid repeating literal values throughout the codebase.

---

# Spacing

Spacing should come from a design scale.

Avoid arbitrary spacing values.

Consistency is more important than precision.

---

# Colors

Colors should be semantic.

Examples:

--color-primary

--color-border

--color-success

Avoid:

--green

--blue2

--gray-darker

---

# Breakpoints

Maintain one breakpoint system.

Avoid component-specific breakpoints.

Mobile-first always.

---

# Accessibility

Every interactive element must have:

accessible name

visible focus

keyboard support

appropriate semantics

Every dialog must:

trap focus

close with Escape

restore focus

announce itself

---

# Theme Editor UX

Group settings logically.

Avoid overwhelming merchants.

Good example:

Cart Drawer

• Enable drawer

• Free shipping threshold

• Show delivery estimate

• Animation

Bad example:

Scattered settings across multiple sections.

---

# Performance

Before adding JavaScript ask:

Can Liquid solve this?

Can CSS solve this?

Can HTML solve this?

Only use JavaScript when interaction requires it.

---

# Progressive Enhancement

Baseline experience should always work.

JavaScript should enhance.

Never replace.

---

# Error Handling

Every network request should:

handle loading

handle success

handle failure

restore UI

allow retry

---

# Analytics

Never hardcode analytics.

Use existing tracking infrastructure.

Avoid duplicate tracking events.

One customer action should produce one analytics event.

---

# Refactoring Expectations

Whenever modifying Shopify functionality:

Read the surrounding implementation first.

Prefer improving existing systems.

Delete obsolete code.

Reduce duplication.

Simplify configuration.

Maintain backwards compatibility.

---

# Definition of Done

A Shopify feature is complete only when:

✓ Theme Editor compatible

✓ Dynamic Sources supported where appropriate

✓ Localized

✓ Accessible

✓ Mobile verified

✓ Desktop verified

✓ AJAX compatible

✓ No duplicated Liquid

✓ No duplicated JavaScript

✓ No duplicated CSS

✓ App compatible

✓ Search & Discovery compatible

✓ Selling Plan compatible

✓ Performance maintained or improved

✓ Dead code removed

---

# Final Rule

If Shopify already provides a robust solution, use it.

The best Shopify themes feel native to Shopify.

The goal of this repository is to extend the platform—not fight it.
